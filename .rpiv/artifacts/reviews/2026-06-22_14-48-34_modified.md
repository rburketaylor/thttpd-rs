## 🟡 Important

### Q1 🟡 CGI `read_to_end` loads entire output into memory with no bound

**Where**
`rust/crates/thttpd-core/src/eventloop.rs` — `dispatch_cgi`, CGI output collection

**Code**
```rust
let mut output = Vec::new();
if let Some(stdout) = cgi_result.child.stdout.take() {
    let mut stdout = stdout;
    let _ = stdout.read_to_end(&mut output);
}
```

**Why**
A hung or output-heavy CGI program can exhaust server memory before the response reaches the client. C thttpd passes the raw file descriptor and uses kernel pipe buffering — the CGI output streams directly to the client socket without a memory intermediary. This is acknowledged in `docs/KNOWN_DEVIATIONS.md` under "CGI resource control" but the memory-unboundedness is a distinct OOM vector from just timeout/missing-cgilimit.

**Fix**
Cap CGI output to a configurable limit (matching legacy `-c`/`cgilimit` semantics) and stream bytes to the response buffer incrementally instead of collecting the full output first.

**Alt**
Implement a pipe-to-socket streaming path that reads CGI stdout into a fixed-size ring buffer and writes to the client fd without holding the full output in memory.

### Q2 🟡 `daemonize()` double-fork calls `unsafe { fork() }` with no SAFETY comment

**Where**
`rust/crates/thttpd-core/src/startup.rs:251` and `:261`

**Code**
```rust
match unsafe { fork() }.map_err(|e| format!("first fork: {e}"))? {
    ForkResult::Parent { child } => {
        let _ = child;
        std::process::exit(0);
    }
    ForkResult::Child => {}
}
```

**Why**
`nix::unistd::fork()` is `unsafe` because forking in a multi-threaded program can deadlock (only the calling thread survives; held locks are never released). The function is called during single-threaded startup before signal handlers or I/O — making it sound — but without a `// SAFETY:` comment, future maintainers cannot verify that the invariant holds. If a background thread or async runtime is added before daemonization, this becomes a latent deadlock.

**Fix**
Insert a `// SAFETY:` comment above each `unsafe { fork() }` stating that `daemonize()` is called before any threads are spawned, signal handlers are installed, or the event loop starts — the process is strictly single-threaded at this point.

### Q3 🔵 Periodic throttle wake recomputes fair-share and re-registers without calling `handle_send`

**Where**
`rust/crates/thttpd-core/src/eventloop.rs:162` — periodic throttle update loop

**Code**
```rust
slot.state = ConnState::Sending;
slot.throttle.pause_until = None;
let token = conn_token(slab_key);
if let Some(ref mut stream) = slot.stream {
    let _ = server.poll.registry().reregister(
        stream, token, Interest::WRITABLE,
    );
}
```

**Fix**
Add a comment explaining that the WRITABLE re-registration is sufficient — the next poll iteration will dispatch a writable event and route through `handle_connection_event` → `handle_send`. The one-iteration delay is bounded by the poll timeout (already computed from `next_pause_deadline`).

### G2 🔵 `main.rs` adds 84 lines of startup-ordering logic with 0 new tests

**Where**
`rust/crates/thttpd-core/src/main.rs:11–140` — startup sequence

**Code**
```rust
let args = thttpd_core::config::normalize_legacy_args(raw);
let legacy_toggle_events = thttpd_core::config::legacy_toggle_events(&args);
```

**Why**
The startup ordering (logfile → pidfile → daemonize → chroot → data_dir → bind → setuid → throttle install) is security-critical. Incorrect ordering could bind before chroot (serving files outside the jail) or drop privileges before writing the pidfile. The existing integration/differential tests exercise the full server but do not isolate startup-ordering regressions.

**Fix**
Add a startup-order integration test that asserts `ServerConfig.dir` is rewritten to `/` after chroot, that listeners bind after chroot, and that the throttle table is installed on the `Server` before `eventloop::run`.

## 💭 Discussion

### Q4 💭 `no_symlink_check` feature bypasses path-traversal protection — is this the right default?

**Where**
`rust/crates/thttpd-core/src/eventloop.rs` — `serve_static`

**Code**
```rust
if server.config.no_symlink_check {
    file_path.to_path_buf()
} else {
    let canonical_root = match std::fs::canonicalize(&server.config.dir) {
```

**Why**
The `no_symlink_check` flag (enabled by `-nos`, `--no-symlinks`, or implied by `-r`/`--chroot`) skips the `canonicalize` symlink-escape guard that prevents serving files outside the document root. This mirrors C thttpd behavior exactly. However, the Rust port has the opportunity to document *why* this exists (performance? specific configurations?) and whether removing the feature outright (always checking) would break any legitimate use case. In a chroot jail, the symlink check is technically redundant since the jail bounds the filesystem — so `-r` implying `no_symlink_check` is sound. The standalone flag (`-nos`) is the risky path.
