# Security Comparison Report + Security CI — Implementation Plan

## Overview

This plan produces **empirical evidence** that thttpd-rs is meaningfully more secure than the C thttpd it replaces, and bakes the evidence-generating machinery into CI so the property holds going forward.

The deliverable is a public-facing report at `docs/security/MIGRATION_REPORT.md` that does three things:

1. Lists every CVE ever filed against `sthttpd` / `thttpd` / `acme thttpd`, mapped to CWE class
2. For each CWE class, identifies the specific C code that exhibited the vulnerability and the specific Rust mechanism that prevents it
3. Documents the runtime mitigations (Miri, sanitizers, `cargo-audit`, `cargo-deny`, `cargo-geiger`) that run in CI on every commit

This is the "Rust is safer" claim with receipts — the artifact that closes the security question in any migration interview.

### What "secure" means here

We make four testable claims:

1. **No known CVEs apply to the Rust port** — verified by enumerating historical CVEs and confirming each is closed in our code
2. **No new unsafe code is added** — verified by `cargo-geiger` in CI, gated to zero `unsafe` in our own code (transitive deps allowed but tracked)
3. **The hot parsers are sound** — verified by Miri on nightly, `cargo-fuzz` smoke test, and `RUSTFLAGS="-Z sanitizer=address"` in CI
4. **Dependency supply chain is clean** — verified by `cargo-audit` (RustSec) and `cargo-deny` (licenses, duplicates) in CI

The report's "Verifiable Claims" section links each claim to the CI job that enforces it.

- `unsafe-audit` job: `cargo geiger --workspace --output-format Json` + threshold check (runs on every PR)

- `sbom` job: `cargo auditable build --release` + `syft scan` + CycloneDX SBOM attached to release (runs on tag)

## What We're NOT Doing

- **Formal verification** — out of scope; Miri + fuzz is the right cost/benefit
- **Penetration testing** — out of scope for this plan; the report is the prerequisite
- **CVSS scoring of hypothetical Rust CVEs** — only historical CVEs are scored
- **Compliance certifications** (SOC2, FedRAMP) — out of scope
- **WASM or sandboxing the server** — out of scope
- **Modifying the C source for hardening** — the C is upstream; we are documenting, not patching
- **Tracking non-sthttpd HTTP server CVEs** (nginx, apache) — out of scope; methodology would extend if needed
- **Claims about safety of transitive dependencies** — tracked via `cargo-audit`, not vouched for in prose

## Phase 3: Rust-side mitigation analysis

### Overview
For each CWE class in Phase 2, identify the specific Rust mechanism that prevents it in `thttpd-rs`. The output is a "CWE → Rust mechanism → file:line" map. This is the "Rust is safer" claim with evidence.

### Changes Required:

#### 1. Rust-mitigation table
**File**: `docs/security/RUST_MITIGATIONS.md`
**Changes**: NEW.

```markdown
# Rust-side Mitigations

> For each CWE class, the mechanism in `rust/crates/*/src/*.rs` that prevents the class.

| CWE | Rust mechanism | Evidence | Confidence |
|-----|----------------|----------|------------|
| CWE-120 | `Vec<u8>` and `String` grow on demand; `read()` is bounded by `&mut [u8]` length | `rust/crates/thttpd-http/src/parse.rs:67-89` (no `read_to_end` on user-controlled slice) | High |
| CWE-125 | Slice indexing is bounds-checked; no raw pointer arithmetic in the HTTP path | `rust/crates/thttpd-http/src/url.rs:34` (indexing returns `Option`) | High |
| CWE-476 | `Option<T>` in return types; `?` propagates `None`; impossible to read from absent value | `rust/crates/thttpd-http/src/parse.rs:142` (`fn header() -> Option<&str>`) | High |
| CWE-20 | `FromStr` returns `Result`; no implicit sentinel values; negative `Content-Length` filtered to `None` (see `eventloop.rs` bug 2 in JOURNEY.md) | `rust/crates/thttpd-core/src/eventloop.rs:201` | High |
| CWE-787 | No `memcpy` of request data; request parsed incrementally through FSM | `rust/crates/thttpd-http/src/parse_state.rs:23-78` | High |
| ... |
```

**Confidence**:
- *High* — the CWE class is structurally impossible in this code (e.g. no buffer to overflow)
- *Medium* — possible only via `unsafe` code, and `cargo geiger` reports zero `unsafe` in our crates
- *Low* — possible only via a logic bug; no compile-time or borrow-checker guarantee

#### 2. `unsafe` audit
**File**: `pipeline/audit_unsafe.sh`
**Changes**: NEW — runs `cargo geiger` and asserts zero `unsafe` in our own crates.

```bash
#!/usr/bin/env bash
set -euo pipefail
cargo install cargo-geiger --locked
cargo geiger --workspace --output-format Json > geiger.json
OURS=$(python3 -c "
import json
data = json.load(open('geiger.json'))
# count unsafe in our packages only (not in deps)
unsafe = sum(p['unsafe']['functions'] for p in data['packages']
             if p['name'].startswith('thttpd'))
print(unsafe)
")
if [ "$OURS" -gt 0 ]; then
  echo "FAIL: $OURS unsafe functions in thttpd-rs (must be 0)"
  exit 1
fi
echo "PASS: 0 unsafe in thttpd-rs"
```

  schedule:
    - cron: '0 4 * * *'  # 4am UTC nightly
  workflow_dispatch:

    runs-on: ubuntu-latest

        run: cargo +nightly miri test -p thttpd-http --lib

        run: cargo +nightly miri test -p thttpd-tdate --lib

### What we measure

## Testing Strategy

### Automated
- **CI jobs** (Phases 4-5): `security-audit`, `miri`, `sanitizers`, `fuzz-smoke`, `unsafe-audit` — run on schedule + PR
- **Local equivalent** (`pipeline/run_security_scan.sh`): devs run before pushing
- **Validation** (Phase 6): `markdown-link-check`, all claims in the report are testable

### Manual
- **Quarterly CVE review**: re-run `pipeline/refresh_cve_inventory.sh`; diff against the current `CVE_TABLE.md`; if a new CVE is filed against sthttpd/thttpd, update the report
- **External review**: once a year, invite a security researcher to do a paid code review; findings feed back into the report

## Performance Considerations

- **Miri is slow**: ~5min for thttpd-http tests. Acceptable on nightly; PRs run a smaller subset.
- **ASan adds memory overhead**: ~2x memory; tests use 30s timeout. Acceptable.
- **`cargo geiger` is fast**: <30s for the whole workspace.
- **`cargo audit` is fast**: <30s. DB updates add <1m.
- **Total CI overhead**: ~15 minutes nightly, ~3 minutes per PR (excluding the existing test suite).
